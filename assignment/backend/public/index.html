Server and Database

Short description of installation and usage of the backend.

Installation

npm install

Running Server

npm start

Server listens on port 3001 You can define a different port inside nodemon.json file.

Reset Database

Database starts out empty but initial set of users and products can be saved to the database with the following command: 

npm run reset-db

Database is saved to db/db.json and it can be reset again with the above command. If you want to start with empty database just delete the file db/db.json

Communication with the Backend

Description of the communication between frontend and backend.

Request Headers

Every request which expects a JSON response (i.e. GET, DELETE, POST, and PUT) needs to specify a header: Accept: application/json and every request which sends JSON data to the backend (i.e. POST and PUT) needs to also specify the header: Content-Type: application/json

Authentication

Server uses cookies and JWT tokens for user authentication and expects the token to be delivered inside a cookie. The cookie is set to be http-only which means that the cookie (and JWT token) are inaccessible from JavaScript. The browser should send the cookie automatically with every subsequent request to the backend.

The lifetime of the cookie is set to 3600 seconds (one hour) but it is refreshed every time a new request is made to the backend.

The backend sets the cookie automatically after a successful login and/or registration. This means that a separate login is not needed after a successful registeration of a new user.

In addition to login and registration there is a special endpoint for the frontend app to check the currently logged in user's data (and role). If there is a cookie present in the request the endpoint validates the JWT token and returns logged in user's data in a JSON object. Othewise the endpoint returns an object which contains only the role (guest) of the user.

Whenever a logged in (i.e. known) user is trying to access a route which is not allowed for that user the server responds with a status code 403. For guests (i.e. unknown users) trying to access routes which require authentication the response code is 401.

See:

  • POST /api/register
  • POST /api/login
  • GET /api/check-status

Error Responses

Endpoints which require JSON data inside the request body will validate the supplied data and respond with status 400 and an error object containing a detailed reason for the error. A status of 404 is also possible if the endpoint requires a URL parameter such as productId, userId, or orderId. Then the error object will contain a message informing that an object with the specified ID could not be found. All of these errors are part of the normal operation and the frontend should account for them.

Other possible errors are related to missing headers or other more serious errors. A status of 406 indicates a missing or invalid Accept header and a status of 415 indicates a missing or invalid Content-Type header. Unrecoverable or unknown server errors result in status 500.

See: Error

Check user's current login status and role. For logged in users who supply a valid cookie containing a valid JWT token the user object is returned without the password field. Guests and users whose cookie or token is invalid or timed out the returned object contains only the role field.

Path Parameters

None

Request Body

Empty

Responses

200
User current loggend in user

  }
}

Content-Type: application/json

    "role": "guest"
